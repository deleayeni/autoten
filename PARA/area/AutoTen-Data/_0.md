# AutoTen Data — Area

## What this Area tracks

The **data layer** of AutoTen: Supabase schema (what tables exist, what columns), RLS policies (who can read/write what), data flow (how a row gets from Postgres to a React component), and Storage (where images live).

This is the answer to: "what does the database look like right now, and how does the app talk to it?"

## Why this is separate from AutoTen-Frontend

Frontend cares about "what the user sees". Data cares about "what we have in Postgres". These evolve independently — you can redesign a page without touching the schema, or add a column without touching UI.

## What I know vs. what needs verification

**Known from the code:**
- The `cars` table exists and has specific columns (see [cars-table.md](./cars-table.md))
- The site queries with `.eq("published", true).eq("status", "available")` — so those columns must exist and RLS must permit SELECT for `anon`
- The type contract is [src/types/car.ts](../../../src/types/car.ts)

**Needs verification in the Supabase dashboard:**
- Is RLS actually enabled on `cars`?
- What's the exact SELECT policy text? (USING expression, target roles)
- Are there other tables already (leads, bookings) that aren't visible from code?
- What Storage buckets exist? Are they public?

When you next open the Supabase dashboard, verify and update the relevant files here.
